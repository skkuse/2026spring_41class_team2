# MovieLens Subset 생성 및 DB 저장 계획

## MovieLens Subset이 필요한 이유

MovieLens Subset은 MovieLens 전체 데이터에서 Cinemate가 실제로 서비스할 영화만 골라낸 작은 추천용 데이터셋이다. 이 subset은 Cinemate의 `movies` 테이블에 저장할 영화 3,000개를 정하는 기준이 된다.

MovieLens 원본에는 영화 수가 많고 추천 계산에 필요한 태그 점수도 매우 크기 때문에, 전체 데이터를 그대로 저장하지 않고 서비스에 사용할 영화만 먼저 추려야 한다. 특히 Cinemate는 Supabase Free tier를 사용하므로, DB 용량 제한 안에서 영화 데이터와 추천 검색 데이터를 함께 저장할 수 있도록 subset을 만들어야 한다.

MovieLens 주요 파일과 Tag Genome 설명은 [movielens-dataset.md](./movielens-dataset.md)에서 다룬다.

## 목표

MovieLens에서 추천 품질이 좋은 영화 3,000개를 선정하고, 이 영화들을 기준으로 Cinemate DB를 채운다.

| 항목 | 결정 |
|---|---|
| 최종 저장 영화 수 | 3,000개 |
| DB 용량 목표 | Supabase 전체 사용량 약 400MB 이하 |
| 영화별 Tag Genome 점수 | relevance 상위 50개 |
| 예상 tag score row 수 | 150,000개 |
| 장르 기준 | MovieLens 장르 무시, TMDB 장르 사용 |

최종 3,000개는 반드시 보장한다. TMDB 조회 실패, poster/overview 누락, `tmdbId` 누락 등으로 제외되는 영화가 생기면 다음 순위 영화로 backfill한다.

## 영화 선정 기준

기본 후보는 다음 조건을 모두 만족해야 한다.

```text
Tag Genome에 존재한다.
links.csv에 tmdbId가 존재한다.
tmdbId가 중복되지 않는다.
rating_count >= 500
user_tag_count >= 20
poster_path가 존재한다.
ko-KR 또는 en-US overview가 존재한다.
```

후보 영화는 다음 점수로 정렬한다.

```text
selection_score =
  log(rating_count + 1) * 0.6
  + log(user_tag_count + 1) * 0.3
  + average_rating * 0.1
```

평점 수는 대중적 신뢰도, 사용자 태그 수는 메타데이터 풍부함, 평균 평점은 품질 보정 신호로 사용한다.

## TMDB 데이터 보강

MovieLens에는 Cinemate 화면에 필요한 포스터, 줄거리, 상세 정보가 충분하지 않다.

따라서 최종 선정된 영화의 `tmdbId`로 TMDB API를 호출해 화면 표시용 영화 정보를 보강한다. 한국어 정보를 우선 사용하고, 없으면 영어 정보로 fallback한다.

화면별 API 요구사항 기준으로 TMDB에서 보강할 필드는 다음과 같다.

| 필드 | 사용 목적 |
|---|---|
| `id` | TMDB movie id. Cinemate의 movie id로 사용한다. |
| `title` | 영화 카드, 검색 결과, 상세 화면에 표시할 영화 제목 |
| `original_title` | 상세 화면의 원제 표시 |
| `overview` | 상세 화면의 `synopsis`, LLM 추천 이유 생성에 사용 |
| `poster_path` | 영화 카드, 검색 결과, 찜 목록, 추천 결과에 표시할 포스터 이미지 |
| `backdrop_path` | 영화 상세 화면의 배경 이미지 |
| `release_date` | MovieCard와 상세 화면의 개봉 연도 `year` 계산 |
| `runtime` | 영화 상세 화면의 러닝타임 표시 |
| `original_language` | 영화 상세 화면의 원어 정보 표시 |
| `production_countries` | 영화 상세 화면의 제작 국가 `countries` 표시 |
| `genres` | Cinemate 장르 테이블과 영화-장르 연결 데이터 생성 |
| `credits.crew` | 영화 상세 화면의 `director` 추출 |
| `credits.cast` | 영화 상세 화면의 `cast` 생성 |
| `videos` | 영화 상세 화면의 `trailerUrl` 생성 |

언어와 fallback 정책은 다음과 같다.

```text
ko-KR title, overview 우선 사용
ko-KR 값이 비어 있으면 en-US fallback 사용
poster_path는 언어와 무관하게 사용
genre는 TMDB genre id 기준으로 저장
```

다음 경우에는 해당 영화를 제외하고 다음 순위 영화로 backfill한다.

```text
TMDB 조회 실패
poster_path 없음
ko-KR와 en-US overview 모두 없음
```

## 장르 정책

MovieLens 장르는 사용하지 않는다. Cinemate 서비스의 장르 기준은 TMDB 장르다.

1차 구현에서는 장르별 quota를 강제하지 않는다. 먼저 선정 점수 기준으로 3,000개를 추출한 뒤, TMDB 장르별 영화 수를 확인한다.

장르 분포가 한쪽으로 심하게 치우치면 이후 보정 기준을 다시 논의한다.

## DB 저장 구조

MovieLens subset은 Cinemate 서비스 영화 목록의 기준이 된다.

저장 순서:

```text
1. MovieLens에서 풍부한 영화 후보를 selection_score 순으로 정렬
2. TMDB API로 영화 카드 정보를 보강하면서 최종 3,000개 확보
3. Cinemate movies, genres, movie_genres 테이블에 영화 데이터 저장
```

## 생성할 중간 파일

전처리 스크립트는 Supabase 저장용 CSV를 생성한다.

```text
movies_seed.csv
genres_seed.csv
movie_genres_seed.csv
```

## 구현 단계

1. `ratings.csv`에서 `movieId`별 `rating_count`, `average_rating`을 계산한다.
2. `tags.csv`에서 `movieId`별 `user_tag_count`를 계산한다.
3. `links.csv`에서 `movieId -> tmdbId` 매핑을 만든다.
4. `genome-scores.csv`에 존재하는 MovieLens `movieId` 목록을 구한다.
5. `tmdbId` 없음, 중복 `tmdbId`, `rating_count < 500`, `user_tag_count < 20` 후보를 제외한다.
6. `selection_score`로 후보를 정렬한다.
7. TMDB API로 후보 영화 정보를 조회한다.
8. poster 또는 ko/en overview가 없는 영화는 제외한다.
9. 최종 3,000개가 될 때까지 다음 순위 영화로 backfill한다.
10. 최종 3,000개의 TMDB 장르 분포를 확인한다.
11. `movies_seed.csv`, `genres_seed.csv`, `movie_genres_seed.csv`를 생성한다.
12. Supabase에 테이블을 생성한다.
13. CSV를 Supabase에 저장한다.
14. 생성된 테이블의 row 수, DB 용량, 기본 조회 가능 여부를 검증한다.
