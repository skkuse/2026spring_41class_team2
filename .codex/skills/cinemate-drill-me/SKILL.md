# Cinemate Drill Me

## 목적

기능 구현 전에 `docs/implementation-plan/**`, `docs/api-spec/**`, `docs/db-schema/**`, 관련 코드 구조와 테스트 패턴을 기준으로 사용자와 Codex의 이해를 맞춘다.

이 skill은 구현하지 않는다. 파일 수정, 테스트 작성, 리팩터링을 하지 않는다.

## 진행 흐름

1. 관련 `docs/implementation-plan/**`, `docs/api-spec/**`, `docs/db-schema/**`를 읽는다.
2. `AGENTS.md`, 관련 코드 구조, 기존 테스트 패턴을 확인한다.
3. Codex가 이해한 기능 범위와 주요 결정사항을 짧게 요약한다.
4. 문서, 코드 구조, 테스트 패턴, 사용자 답변 사이에 충돌하거나 애매한 점이 있으면 질문한다.
5. 답변을 반영해 다시 요약한다.
6. 사용자가 이해가 일치했다고 확인하면 다음 개발 단계에 넘길 합의 내용을 정리한다.

## 질문하지 않는 것

- 프론트엔드 구현 세부사항
- 버튼, 레이아웃, 컴포넌트 분리 같은 UI 세부 결정
- 기존 코드 패턴으로 자연스럽게 결정 가능한 사항
- `AGENTS.md`에 이미 정해진 프로젝트 규칙
- 패키지 매니저, App Router, 테스트 파일 위치 같은 고정 규칙

프론트엔드 세부 구현은 Codex가 기존 UI/UX 패턴, 접근성, 반응형 기준에 맞춰 합리적으로 결정한다.

## 출력 형식

합의 전에는 다음 형식으로 응답한다.

```text
이해한 내용

- ...

확인할 질문

1. ...
2. ...
```

합의 후에는 다음 형식으로 정리한다.

```text
합의된 개발 입력

확정된 요구사항:
- ...

확정된 데이터/API/도메인 결정:
- ...

제외 범위:
- ...
```

## 합의 후 동작

사용자가 "진행", "구현해줘", "이대로 개발", "ㄱㄱ"처럼 개발 시작 의사를 명시하면, 합의된 개발 입력을 기준으로 `cinemate-feature-dev` skill을 사용해 구현을 시작한다.

사용자가 개발 시작 의사를 명시하지 않으면 구현하지 않고, 합의된 개발 입력만 정리한다.

## 주의사항

- 코드베이스에서 확인 가능한 것은 먼저 확인한다.
- 사용자가 이미 결정한 내용은 다시 묻지 않는다.
- 합의된 내용은 다음 개발 skill이 바로 사용할 수 있게 명확히 정리한다.
